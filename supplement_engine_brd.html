Technical Business Requirements Document

Personalized Supplement
Recommendation Engine

A hybrid symbolic-statistical clinical decision support engine that reads normalized patient state from PostgreSQL and clinical evidence from Neo4j, then produces ranked nutraceutical recommendations through a deterministic safety gate. Patient bulk backfill is owned by an external feeder project (writes Postgres directly); this engine handles realtime deltas (labs, meds, conditions) and scoring — never warehouse joins at request time.

Status: Phase 2a–2c M1/M2 shipped
Next: External feeder integration + clinical sign-off
Classification: CDSS-Adjacent / Wellness
Region: Saudi Arabia (KSA) primary
Version: 1.1
Stack: Python 3.12 · Neo4j · PostgreSQL · FastAPI · Redis
Section 01

System Architecture

The system has three layers: ingest (external bulk feeder → Postgres + this engine’s delta APIs), stores (PostgreSQL patient realm + Neo4j knowledge graph), and online scoring (stateless recommendation pipeline + deterministic safety gate). The Recommender Core reads typed domain objects only; vocabulary mapping happens in Intake or the external feeder.

Non-negotiable split: Neo4j owns clinical knowledge. PostgreSQL owns patient PHI, sessions, audit, and feedback. They link by vocabulary keys (icd10_code, rxnorm_cui, loinc, nutrient_id) through the application layer only. No direct DB-to-DB connection.

Intake & Normalizer

Shared normalization for bulk ETL and realtime APIs: RxNorm, ICD-10, LOINC, unit conversion. Writes patient realm in Postgres. Phase 2.

Recommender Core

Bayesian DRS + candidate generation + dose optimization + confidence. Reads PatientProfile from Postgres + Neo4j edges. Phase 1 shipped.

Safety Engine

Deterministic Python gate: drug-nutrient, disease CI, antagonists, 70% UL cap, escalation. No ML. Runs after recommender. Phase 1 shipped.

Personalization ML

Bayesian individual update, dose optimization, lab integration. SLO: p95 < 400ms

Evidence Service

Returns GRADE-rated citations per recommendation claim. SLO: p95 < 100ms

Explain Service

Template-driven rationale (Phase 1) + optional constrained LLM polish (Phase 2). SLO: p95 < 600ms

Section 01c

Data Ingestion Architecture

The patient realm in PostgreSQL is fed by two ingress paths: an external bulk feeder (separate project writing directly to patient tables) and dynamic intake APIs in this engine (labs, meds, conditions). The recommendation engine consumes a single contract: frozen PatientProfile objects loaded via repositories. It never queries an upstream warehouse or performs EHR joins at request time.

flowchart LR subgraph sources [External Sources] WH[Existing source system / warehouse] EHR[EHR / LIS / FHIR] end subgraph ingest [Ingest Layer - Phase 2] ETL[Airflow bulk ETL] API[Intake APIs] NORM[IntakeNormalizer] end subgraph stores [Canonical Stores] STG[(staging.* raw)] PG[(Postgres patient realm)] NEO[(Neo4j KG)] end subgraph online [Online Scoring - Phase 1 shipped] REC[POST /v1/recommendations] PIPE[RecommendationPipeline] end WH --> ETL --> STG --> NORM --> PG EHR --> API --> NORM ETL --> NEO REC --> PG REC --> PIPE PIPE --> PG PIPE --> NEO

Path A: Bulk ETL (existing source system)

For initial backfill and nightly refresh from your existing database (Postgres, SQL Server, Impala, or warehouse). Joins and preprocessing run in Airflow/dbt, not in the API.

📦

Extract

Pull demographics, conditions, meds, labs from source DB

🛠

Stage

Load into staging.* with source_system, batch_id, extracted_at

🔄

Normalize

Shared IntakeNormalizer: RxNorm, ICD-10, LOINC, units

👤

Identity

hashed_mrn = SHA-256(source_system:mrn) maps to patient_id UUID

💾

Merge

Upsert into patients, conditions, meds; append labs

Path B: Dynamic intake (realtime APIs)

For intraday deltas: new lab results, medication changes, FHIR webhooks. Same normalizer, same Postgres tables, same merge rules.

Merge semantics (bulk + API must agree)

Phase 1 PatientRepository.upsert() replaces all child records on update. Phase 2 introduces merge_from_source() with source-aware rules so bulk nightly loads and intraday API writes do not clobber each other.

EntityMerge ruleNotes
DemographicsLatest wins by source priority or updated_atbmi, pregnancy_status, region_code, lifestyle flags
ConditionsUpsert by (patient_id, icd10_code, onset_date)source: warehouse / ehr / self / clinician
MedicationsUpsert by (patient_id, rxnorm_cui, start_date)Mark stopped if absent from bulk snapshot
LabsAppend onlyKey: (patient_id, loinc, collected_at); never delete history on refresh

Knowledge graph ingestion (separate from patient realm)

Neo4j is fed by batch processes only: initial neo4j_seed.cypher (v1.1.0 shipped), then weekly evidence Airflow DAG (Phase 2). Patient ETL never writes clinical LR edges directly without curator approval.

User Stories

Patient (Consumer)

US-01As a patient with T2DM and a new metformin prescription

I want the engine to automatically recognize that metformin depletes B12 and recommend a monitored supplement, so that I don't develop a deficiency without knowing.

  • Given: patient_medication has rxnorm_cui for metformin with months_on ≥ 3
  • When: evaluate() is called
  • Then: B12 candidate appears with confidence ≥ 0.80 and rationale citing metformin depletion
US-02As a pregnant patient in Riyadh

I want folate, iron, iodine, choline, and DHA forced into my recommendations at guideline doses, and high-dose retinol blocked, so that my fetal development is protected.

  • Given: pregnancy_status = true, region_code = SA-01
  • Then: ACOG/WHO pregnancy guideline overrides fire; retinol > 3000 µg RAE blocked absolutely
US-03As a patient with CKD Stage 4

I want potassium and magnesium supplements blocked and a clinician escalation triggered, so I don't inadvertently worsen hyperkalemia.

  • Given: patient_condition icd10_code = N18 with stage ≥ 4
  • Then: K and Mg recommendations suppressed; requires_clinician = true returned in session

Clinician

US-04As an endocrinologist reviewing a patient dashboard

I want to see the full evidence trace for each recommendation — GRADE rating, sample size, and source guideline — so I can decide whether to accept, modify, or reject it.

  • GET /v1/evidence/{snapshot_id} returns KG snapshot within p95 < 100ms
  • Each session links to evidence_snapshot_id (immutable KG state at time of recommendation)
US-05As a clinician overriding an AI recommendation

I want my override captured in the feedback loop so it retrains the Bayesian priors for similar patient profiles over time.

  • POST /v1/feedback with action = "rejected" or "modified" + notes
  • Override feeds nightly retraining pipeline → next week's priors shift

Data / ML Engineer

US-06As an ML engineer running a new rule version

I want to create a new rule from a JSON definition and immediately evaluate it against a patient cohort, getting performance metrics vs the previous version.

  • python cli.py create --input-file rule.json → returns rule_version_id
  • python cli.py execute {rule_version_id} → runs OptimizedRuleEngine; PerformanceBenchmark compares
US-07As a data analyst building a deficiency prevalence map

I want a de-identified aggregate view of which nutrients are most flagged per ICD-10 condition per region, so the public health team can target interventions.

  • OLAP mart: fact_recommendations × dim_condition × dim_region → prevalence cube
  • Anonymized: no patient_id in mart; aggregated at bucket_id level (age/sex/region)

System / Compliance

US-08As a compliance officer preparing for SFDA SaMD audit

I want every recommendation output to be reproducible from its evidence_snapshot_id and model_version, with a full audit trail, so regulators can reconstruct any historical decision.

  • evidence_snapshots table stores KG commit SHA + JSONB contents at serve time
  • Audit log: append-only, 7-year retention, includes input hash, model version, output diff

Data Engineering / Ingest

US-09As a data engineer syncing patients from our existing warehouse

I want nightly bulk ETL to load demographics, conditions, medications, and labs into the patient realm using the same normalized schema as realtime APIs, so the recommendation engine can score by patient_id without duplicating ingest logic.

  • Given: Airflow dag_patient_bulk_sync extracts from source DB
  • When: rows are normalized and merged via hashed_mrn
  • Then: POST /v1/recommendations with patient_id returns recommendations using bulk-loaded data
US-10As a lab interface sending a new Vit D result mid-day

I want to append a lab via the intake API before the next bulk run, so recommendations reflect today's result without waiting for nightly ETL.

  • Given: patient exists from bulk load
  • When: POST /v1/patients/{id}/labs/ingest with LOINC 1989-3
  • Then: patient_labs appended; subsequent recommendation uses new value
